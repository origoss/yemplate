1 Introduction

We at origoss love Kubernetes. Sometimes we use our Kubernetes deployment to run some batch jobs.

Batch jobs are awesome but sometimes we find it a bit hard to work with.

2 Use case

Let's see a bit convoluted use case where we want to copy a git repository to a persistent volume.

2.1 First approach

First, let's create a manifest file for our job that will mount two volumes and perform the copy between the volumes. 

apiVersion: batch/v1
kind: Job
metadata:
  name: git-copy-1
spec:
  template:
    spec:
      containers:
      - name: git-copier
        image: busybox
        volumeMounts:
        - name: gitrepo
          mountPath: /git
        - name: target
          mountPath: /target
        command:
        - sh
        - -c
        - |
          rm -rf /target/*
          rm -rf /target/.*
          cp -a /git/* /target
          ls -a /target/*/*
      restartPolicy: Never
      volumes:
      - name: gitrepo
        gitRepo:
          repository: 'https://github.com/origoss/yemplate.git'
      - name: target
        persistentVolumeClaim:
          claimName: git-repo-target

kubectl apply -f job1.yaml

job.batch "git-copy-1" created

2.2 Repeat with another git repo

We demonstrated, that it is relatively easy to start a job in Kubernetes. What if we want to repeat the previous exercise using a slightly different parameters, e.g. for a different git repository?

For jobs, the current standard solution is to create a new yaml file based on the one shown in Listing 1.

If we perform such a task regularly this task cries for automation. If it was a deployment, we could use helm or some other available tool to create a reusable template.

Fortunately, for jobs, we can use the simple yemplate tool.

Let's create a job template based on the previous job manifest. 

 1: apiVersion: batch/v1
 2: kind: Job
 3: metadata:
 4:   name: git-copy-1
 5: spec:
 6:   template:
 7:     spec:
 8:       containers:
 9:       - name: git-copier
10:         image: busybox
11:         volumeMounts:
12:         - name: gitrepo
13:           mountPath: /git
14:         - name: target
15:           mountPath: /target
16:         command:
17:         - sh
18:         - -c
19:         - |
20:           rm -rf /target/*
21:           rm -rf /target/.*
22:           cp -a /git/* /target
23:           ls -a /target/*/*
24:       restartPolicy: Never
25:       volumes:
26:       - name: gitrepo
27:         gitRepo:
28:           repository: {{ .repository }}
29:       - name: target
30:         persistentVolumeClaim:
31:           claimName: git-repo-target

The only difference (besides the meaningless .t extension) is the repository field that had been replaced with a template markup {{ .repository }} in line 28.

We can use yemplate now to fill in the holes: 

yemplate job1.yaml.t                                              \
         --set repository=https://github.com/origoss/yemplate.git

apiVersion: batch/v1
kind: Job
metadata:
  name: git-copy-1
spec:
  template:
    spec:
      containers:
      - name: git-copier
        image: busybox
        volumeMounts:
        - name: gitrepo
          mountPath: /git
        - name: target
          mountPath: /target
        command:
        - sh
        - -c
        - |
          rm -rf /target/*
          rm -rf /target/.*
          cp -a /git/* /target
          ls -a /target/*/*
      restartPolicy: Never
      volumes:
      - name: gitrepo
        gitRepo:
          repository: https://github.com/origoss/yemplate.git
      - name: target
        persistentVolumeClaim:
          claimName: git-repo-target

As you can see, yemplate generated to the stdout a proper Kubernetes job manifest file with the provided parameter in the value of the repository field.

We can use this for creating our job: 

yemplate job1.yaml.t                                                \
         --set repository=https://github.com/origoss/yemplate.git | \
    kubectl apply -f -

job.batch "git-copy-1" created
